# Memory Exhaustion in `Box_snuc::parse()` Bypasses Security Pixel Limit

## Summary

`Box_snuc::parse()` allocates two `std::vector<float>` sized to
`image_width * image_height` pixels. A crafted snuc box with `image_width=2`,
`image_height=117,966,856` produces 235M pixels — within the 1G-pixel security
limit — but the two `std::vector::resize()` calls together allocate ~1.88 GB,
exhausting memory. The allocations bypass the `MemoryHandle` tracking system used
by other box parsers.

- **Affected file:** `libheif/codecs/uncompressed/unc_boxes.cc`, lines 1318/1323/1328
- **Type:** CWE-400 Uncontrolled Resource Consumption
- **Severity:** Medium — DoS via malicious HEIF file; 61-byte POC triggers OOM
- **Affected version:** libheif 1.21.2 (current HEAD)

---

## Vulnerability Details

### Root Cause

`Box_snuc::parse()` in `unc_boxes.cc` checks the pixel count against the security
limit and then calls `m_memory_handle.alloc()` to register the expected allocation:

```cpp
// unc_boxes.cc ~line 1303
uint64_t num_pixels = static_cast<uint64_t>(m_nuc.image_width) * m_nuc.image_height;

// Check 1: pixel count limit (passes for 235M pixels < 1G)
if (limits->max_image_size_pixels && num_pixels > limits->max_image_size_pixels) {
    return error("snuc image dimensions exceed security limit.");
}

// Check 2: register expected allocation in MemoryHandle (1.88 GB)
Error err = m_memory_handle.alloc(2 * sizeof(float) * num_pixels, limits, "snuc box");
if (err) { return err; }   // Would fail if max_total_memory < 1.88 GB

// BUG: These resize() calls make independent OS allocations NOT tracked
m_nuc.nuc_gains.resize(num_pixels);    // line 1323: malloc(943,734,848 bytes)
m_nuc.nuc_offsets.resize(num_pixels);  // line 1328: malloc(943,734,848 bytes)
```

The `MemoryHandle` system only **tracks** declared usage; it does not allocate the
memory. The actual allocations are made by `std::vector::resize()` via the system
allocator. These allocations are independent of and additive to the MemoryHandle
tracking.

### Memory Accounting

For `image_width=2`, `image_height=117,966,856`:

| Allocation | Size | Tracker |
|-----------|------|---------|
| `m_memory_handle.alloc(2*4*num_pixels)` | 1,887,469,696 bytes (1.88 GB) | Tracked (but no actual malloc) |
| `nuc_gains.resize(num_pixels)` | 943,734,848 bytes (0.94 GB) | **Untracked** — real malloc |
| `nuc_offsets.resize(num_pixels)` | 943,734,848 bytes (0.94 GB) | **Untracked** — real malloc |
| **Total OS request** | **~3.78 GB** | — |

Even if `max_total_memory` is set to 2 GB (as in `box_fuzzer.cc`), the
`m_memory_handle.alloc()` call succeeds (1.88 GB < 2 GB), and then the two
`resize()` calls each request ~944 MB from the OS, causing OOM.

### Design Flaw

The `MemoryHandle` system exists precisely to prevent excessive OS allocations, but
it only works when the tracked amount matches the actual allocation amount. Here the
tracking records 1.88 GB but the actual OS allocations total 1.88 GB **additional**
bytes — the `alloc()` call does not substitute for the `resize()` calls; it merely
registers intent. The two mechanisms are decoupled, making the guard ineffective.

## Impact

A malicious HEIF file containing a single 61-byte `snuc` box can cause the parsing
process to request ~1.88 GB of OS memory regardless of the configured security
limits. Any application using libheif to process untrusted files (image viewers,
web services, media transcoders) is vulnerable to denial-of-service through memory
exhaustion.

- **Attack vector:** Remote/file-based; no authentication required
- **Attack complexity:** Low — single 61-byte box is sufficient
- **Availability impact:** High — process killed by OOM; service disruption

---

## Suggested Fix

Replace the `std::vector::resize()` calls with allocations that are properly
bounded and tracked. The simplest approach is to guard the resize with an explicit
size check and then have the `MemoryHandle` track the actual combined allocation:

```cpp
// unc_boxes.cc, after the m_memory_handle.alloc() check:

// Cap at a reasonable absolute limit before actually allocating
if (num_pixels > 100'000'000ULL) {  // 100M floats = ~381 MB per vector
  return {heif_error_Memory_allocation_error,
          heif_suberror_Security_limit_exceeded,
          "snuc pixel count exceeds safe allocation limit"};
}

m_nuc.nuc_gains.resize(num_pixels);
m_nuc.nuc_offsets.resize(num_pixels);
```

A more principled fix is to restructure `Box_snuc` to read data streaming (without
pre-allocating full vectors), or to use a separate `MemoryHandle` per vector and
ensure the `alloc()` tracks only what the corresponding `resize()` will actually
allocate — then check both allocations sum within `max_total_memory` before either
`resize()` is called.
